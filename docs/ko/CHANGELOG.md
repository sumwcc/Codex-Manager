## [0.1.16] - 2026-04-05

### 추가됨
- `/v1/responses` WebSocket 요청 지원, 전송 유형 식별 완료, 요청 헤더 정규화, 프록시 런타임 및 요청 로그 링크가 추가되었습니다.
- 새로운 추가 할당량 창 표시가 계정 페이지 및 사용량 팝업 창에 추가되었습니다. 새로 고침 후에는 표준 할당량과 코드 리뷰/스파크 등 추가 할당량의 남은 할당량과 재설정 시간이 균일하게 표시됩니다.
- 게이트웨이 추적은 HTTP/WS 원래 요청이 명시적으로 `service_tier`, 원래 값 및 로그 정규화된 값을 전달하는지 여부를 기록하는 새로운 `CLIENT_SERVICE_TIER` 이벤트를 추가하여 클라이언트의 명시적인 `fast`를 플랫폼 키의 기본 서비스 수준과 신속하게 구별할 수 있도록 합니다.

### 수정됨
- HTTP 및 WS 요청 로그에서 `service_tier`의 일관성 없는 구경 문제를 수정했습니다. 이제 `fast`는 클라이언트가 명시적으로 `service_tier`을 전달하도록 요청할 때만 기록되며, 플랫폼 키 기본값은 더 이상 명시적으로 활성화하라는 요청으로 실수로 기록되지 않습니다.
- 로그 페이지의 서비스 수준 표시가 게이트웨이 온와이어 값과 일치하지 않는 문제를 수정했습니다. `priority`은 `fast`으로 일률적으로 표시되며 서비스 수준을 명시적으로 전달하지 않는 요청은 계속해서 `auto`로 표시됩니다.

### 변경됨
- 릴리스 버전은 `0.1.16`으로 업그레이드되며, workspace, 프런트엔드 패키지, Tauri 데스크탑, 잠금 파일, README, CHANGELOG의 버전 노트가 동시에 업데이트됩니다.

- 런처가 콘솔 창을 닫은 후에도 Windows 로컬 Web `codexmanager-service` / `codexmanager-web`이 여전히 백그라운드에 남아 있는 문제를 수정했습니다. the launcher will now recycle the child process together through the Job Object.

- ChatGPT 로그인 계정 요청은 이제 저장된 `refresh_token`로 로컬 `access_token`을 새로 고치고 현재 요청을 한 번 재시도하여 `401`에서 복구됩니다.

### 변경됨
- ChatGPT 로그인 계정은 이제 기본 업스트림 경로에서 직접 `access_token`을 사용하고 더 이상 `api_key_access_token` 의미를 혼합하지 않습니다.
- 이제 합성 게이트웨이 터미널 오류는 기존 추적 및 오류 코드 헤더를 유지하면서 구조화된 OpenAI 스타일 `error.message / error.type / error.code` 페이로드를 반환합니다.

### 수정됨
- `codexmanager-web`의 액세스 비밀번호 세션이 재부팅 후에도 계속 사용될 수 있는 문제를 수정했습니다. Web 프로세스를 닫았다가 다시 열면 이전 로그인 쿠키가 무효화되고 비밀번호를 다시 확인해야 합니다.
- 소스 코드가 `codexmanager-web`을 실행할 때 시작 및 루트 라우팅 호환성 문제를 수정하고 Axum 라우팅에서 Web 정적 리소스 및 루트 경로의 일관되지 않은 동작을 줄입니다.
- 잘못된 판단으로 인해 장기 스트리밍 응답이 중단될 가능성을 줄이기 위해 긴 출력 시나리오에서 SSE 유휴 연결 끊김 및 재연결 문제를 수정했습니다.
- 설정 페이지에서 업스트림 에이전트 저장, 플랫폼 키 생성 팝업 창을 닫고 반복적으로 제출, 로그인 성공 후 계정 양식이 새로 고쳐지지 않는 등 데스크톱 상호 작용 문제를 수정했습니다.
- 모델을 가져올 때 기본 추가 버전 매개변수로 인해 발생하는 일부 업스트림 호환성 문제를 수정하고, 기본적으로 버전 번호를 포함하지 않도록 모델 요청을 변경했습니다.
- 계정 가져오기와 로그인 콜백 간의 계정 병합 로직이 일관되지 않는 문제를 수정하고, 동일한 ID 규칙에 따라 신규 또는 업데이트된 계정을 통합했습니다.
- 여러 MCP 서버 시나리오에 적용할 때 Claude / Anthropic `/v1/messages`에서 도구 잘림 문제를 수정했습니다. 처음 16개의 도구가 가득 차서 후속 서버의 도구가 더 이상 손실되지 않습니다.
- MCP 도구 이름이 너무 길 때 매핑 불안정성을 방지하기 위해 Claude / Anthropic `/v1/messages`에서 긴 도구 이름 단축 및 응답 복원이 누락되는 문제를 수정했습니다.

### 변경됨
- 게이트웨이 실패 응답은 구조화된 `errorCode` / `errorDetail` 필드를 추가하고 동시에 `X-CodexManager-Error-Code`, `X-CodexManager-Trace-Id` 응답 헤더를 추가하여 클라이언트와 로그 시스템이 실패한 링크를 쉽게 추적할 수 있도록 합니다.
- 프로토콜 적응은 계속해서 Codex / OpenAI 호환 생태계를 정렬합니다. 즉, `/v1/chat/completions`, `/v1/responses`, Claude `/v1/messages`의 전달 의미를 더욱 통합하고 `tools` / `tool_calls`, 사고/추론, 스트리밍 브리징 및 응답 복원 링크를 안정화합니다.
- 설정 페이지와 런타임 구성은 계속해서 수렴됩니다. 백그라운드 작업, 게이트웨이 전송, 업스트림 프록시, Web 보안 등과 같은 빈도가 높은 구성은 `app_settings`에 의해 통합되고 지속되며 현재 프로세스에 백필됩니다.
- 데스크톱 및 서비스 시작 링크는 계속 관리되고, Web/service/desktop 사이의 시작 경계와 시작 순서가 통합되며, 소스 코드 실행과 패키지 실행 간의 동작 분기가 줄어듭니다.
- 이 프로젝트는 장기적인 유지 관리 중심의 재구성 관리를 지속적으로 촉진합니다. 프런트 엔드 기본 입구, 설정 페이지, 요청 로그 보기, Tauri 명령 등록, 서비스 수명 주기, 게이트웨이 프로토콜 어댑터, HTTP 브리지, 업스트림 시도 흐름 및 기타 영역은 대규모 파일과 루트 수준 외관 간의 결합을 줄이기 위해 추가로 분할된 모듈 경계를 갖습니다.
- 서비스/게이트웨이 디렉터리 구조는 계속해서 수렴되고 더 많은 와일드카드 가져오기, 계층 간 직접 연결 및 극도로 긴 Facade 목록이 명시적 종속성과 계층화된 모듈로 대체되어 후속 유지 관리 및 프로토콜 회귀 포지셔닝 비용이 낮아졌습니다.
- 릴리스 링크는 계속해서 `release-all.yml` 단일 항목으로 수렴되며 프런트 엔드 구성 제품 및 프로토콜 회귀 기준선은 반복적인 빌드 및 릴리스 중에 프로토콜 회귀 위험을 줄이기 위해 재사용됩니다.
