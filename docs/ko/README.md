# 문서 디렉토리 설명
`docs/`은 CodexManager의 공식 거버넌스 문서 디렉토리이며 더 이상 임시 저장소로 간주되지 않습니다.

목표:
- 구조적 거버넌스, 의사결정 기록, 출고 지시, 운영 매뉴얼 등을 모두 창고의 메인라인에 입력시키세요.
- 새로운 공동 작업자가 구두 전달에 의존하지 않고도 올바른 문서를 찾을 수 있습니다.

## 문서 책임 경계
- 루트 디렉터리 `README.md` / `README.en.md`: 홈 탐색, 프로젝트 개요, 빠른 시작.
- 루트 디렉터리 `변경-이력.md`: 버전 기록 및 출시되지 않은 업데이트에 대한 유일한 정보 소스입니다.
- `report/*`: 작동, 문제 해결, 호환성, FAQ 매뉴얼.
- `release/*`: 빌드, 릴리스, 제품 및 릴리스 프로세스 설명.
- `docs/plan/*` / `docs/decision/*`: 장기 거버넌스, 실행 계획 및 결정 기록.

##최신 업데이트 항목
- 메인 브랜치의 최신 공개 콘텐츠와 미공개 업데이트를 확인하고 싶다면 먼저 [변경-이력.md](변경-이력.md).
- "현재 문제에 가장 적합한 문서가 무엇인지"를 빠르게 판단하고 싶다면 먼저 아래의 "문서 홈 페이지"와 "추천 항목"을 살펴보세요.

## 디렉토리 분업

### `docs/plan/`
구현 계획, 거버넌스 목록 및 단계별 TODO를 저장하는 데 사용됩니다.

### `docs/decision/`
결정 기록 및 ADR을 유지하는 데 사용됩니다.

### `release/`
릴리스 노트, 롤백 계획, 릴리스 승인 기록 및 빌드 릴리스 매뉴얼을 저장하는 데 사용됩니다.

### `report/`
스캔, 문제 해결, 운영, 호환성, FAQ 유형 보고서 및 매뉴얼을 저장하는 데 사용됩니다.

## 규칙 제출

### Git에 커밋해야 하는 문서
- 미래의 공동작업자에게는 여전히 가치가 있습니다.
- 후속 개발, 테스트, 출시 또는 문제 해결 방법에 영향을 미칩니다.
- 프로젝트의 장기적인 정보 소스의 일부로 사용될 수 있습니다.

### 문서를 Git에 제출하는 것은 권장되지 않습니다.
- 임시 초안
- 개인 프로세스 노트
- 일회용 중간제품
- 현지 테스트 기록

## 규칙을 무시하세요
현재 저장소는 다음 문서를 무시합니다.
- `docs/**/*.tmp.md`
- `docs/**/*.local.md`

공식적인 문서인 경우 위의 접미사를 사용하지 마십시오.

## 이름 지정 제안
권장 형식:

``텍스트
장기보존 문서: theme.md
일회성 보고서: yyyyMMddHHmmssfff_topic.md
````

## 유지보수 계약
- 중요한 거버넌스 문서를 추가할 때 `docs/`에 우선순위를 두고 README에 계속 쌓아두지 마세요.
- 버전 기록은 `변경-이력.md`에서 계속됩니다.
- 아키텍처 개요는 `아키텍처.md`에서 계속 유지됩니다.
- 협업 사양은 `기여-가이드.md`에서 계속 유지됩니다.
- 출시되지 않은 업데이트를 여러 개의 긴 문서에 동시에 작성하지 마십시오. 외부 설명이 필요한 경우 `변경-이력.md`을 먼저 추가하면 README에는 요약과 항목만 유지됩니다.
