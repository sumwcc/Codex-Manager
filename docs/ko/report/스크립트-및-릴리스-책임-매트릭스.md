# 대본과 퍼블리싱 책임의 비교

## `.github/actions/`과 `scripts/release/` 비교표

| 시나리오 | GitHub 조치 | 해당 스크립트/책임 |
|---|---|---|
| Tauri 빌드 재시도 | `build-tauri-with-retry` | `scripts/release/build-tauri-with-retry.ps1` / `.sh` |
| Service 포장 | `stage-service-package` | `scripts/release/stage-service-package.ps1` / `.sh` |
| GitHub 출시 | `publish-github-release` | `scripts/release/publish-github-release.sh` |
| 출시 환경 준비 | `setup-release-env` | 워크플로 내부 환경 어셈블리, 현재 완전히 동등한 독립 최상위 스크립트가 없습니다 |
| 프런트엔드 분산 준비 | `prepare-frontend-dist` | 워크플로우 내에서 프런트엔드 제품 엔드포인트 빌드/다운로드, 현재 별도의 최상위 스크립트가 없습니다 |

## 경계 합의

### `.github/actions/`

담당:

- 워크플로 내에서 재사용 가능한 단계
- 작업 전반에 걸쳐 입력과 출력을 통합합니다.
- CI/릴리스 환경 캡슐화

### `scripts/release/`

담당:

- 실제 제작, 패키징, 퍼블리싱 작업
- 로컬로 또는 워크플로별로 호출되는 스크립트 구현
- 독립적으로 실행 가능하도록 유지하십시오.

### `scripts/*.ps1|*.sh`

담당:

- 개발자를 위한 최상위 입구
- 통일된 매개변수 구성
- 복잡한 릴리스 하위 단계를 닫습니다.

## 현재 기록 스크립트 인벤토리

### 여전히 가치 있는

- `rebuild.ps1`
- `bump-version.ps1`
- `tests/gateway_regression_suite.ps1`
- `tests/chat_tools_hit_probe.ps1`
- `tests/codex_stream_probe.ps1`
- `release/*`

이유:

- 여전히 로컬 개발, 릴리스 링크 또는 워크플로에서 실제로 사용됩니다.
- 분리된 레거시 파일이 아님

### 현재 "단일 사용 지점만 있고 직접 삭제할 수 있는" 스크립트는 발견되지 않았습니다.

이 단계에서 더 적절한 전략은 삭제가 아니라 다음과 같습니다.

1. `scripts/README.md`을 사용하여 입구 계층을 나타냅니다.
2. CI 관련 스크립트와 로컬 항목을 별도로 설명
3. 스크립트가 연속해서 여러 버전에서 참조되지 않는 경우 해당 스크립트를 보관하거나 병합하는 것을 고려하세요.

## 유지 관리 권장 사항

- 워크플로 기능을 추가할 때 기존 작업을 재사용할지 여부를 결정하는 데 우선순위를 둡니다.
- 새 릴리스 스크립트를 추가할 때 먼저 "최상위 항목"인지 "CI 하위 단계"인지 결정합니다.
- 스크립트가 로컬 실행에 적합하지 않은 경우 README에 "CI 전용"으로 명확하게 표시되어야 합니다.
