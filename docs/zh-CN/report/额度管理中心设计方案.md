# 额度管理中心设计方案

> 日期：2026-05-11  
> 状态：评审稿  
> 范围：平台级额度管理、上游余额管理、模型价格配置、按模型用量与剩余额度估算

## 1. 背景

当前系统已经具备三块与额度相关的基础能力：

- API Key 侧：平台下发的 API Key 已有 `quotaLimitTokens`，并且请求完成后会写入 `request_token_stats`。
- 聚合 API 侧：本次更新已经加入聚合 API 上游余额查询，支持 `generic` 和 `new_api` 模板，余额结果会写入 `aggregate_apis.last_balance_json`。
- OpenAI / ChatGPT 账号池侧：账号管理已有 usage snapshot、订阅信息、5 小时窗口、7 天窗口和额外 rate limit 窗口。

但这些能力目前分散在 API Key、聚合 API、账号池页面中。用户需要一个独立的“额度管理中心”，统一看清：

1. 自己发出去的 API Key 还有多少额度。
2. 平台上游总共还有多少可用额度。
3. 已用量能否按模型拆分。
4. 剩余额度能否按模型估算。
5. 不同模型、不同供应商的单价如何配置。

本设计的关键原则是：**已用 token 可以按模型精确统计；剩余额度是否能精确到模型，取决于上游是否提供模型级余额。对于未提供模型级余额的上游，只做有标注的估算。**

## 2. 目标与非目标

### 2.1 目标

- 新增一个独立的“额度中心”页面。
- 统一展示 API Key、聚合 API 上游厂商、OpenAI 账号池三类额度来源。
- 支持按模型查看 token 用量、估算成本、预计剩余 token。
- 支持模型价格配置，覆盖 input、cached input、output、reasoning output 等不同 token 类型。
- 支持供应商级价格倍率、余额单位换算、模型价格覆盖。
- 初始化内置一份官方公开价格种子，用户可手动覆盖。
- 所有估算值都要有清晰口径，不能把百分比窗口伪装成精确 token。

### 2.2 非目标

- 不承诺自动抓取所有厂商的实时价格。第一版使用内置价格种子和手动维护。
- 不把 OpenAI / ChatGPT 账号池剩余百分比强行换算为精确 token，除非用户配置计划容量。
- 不把所有来源强行汇总成一个“总 token”。不同来源单位不同，应分别展示。
- 不在前端保存或展示明文上游密钥。

- `request_token_stats` 只有按 key 汇总，没有按 `model`、`key_id + model`、`aggregate_api_id + model` 汇总。
- 模型价格需要抽出为可配置、可复用模块。
- 聚合 API 余额需要接入统一额度来源。
- OpenAI 账号池需要作为一种窗口型额度来源纳入统一页面。

## 4. 核心概念

### 4.1 额度来源 Quota Source

额度中心统一把所有可用额度抽象为 `QuotaSource`。

```ts
type QuotaSourceKind =
  | "api_key"
  | "aggregate_api"
  | "openai_account";

type QuotaMetricKind =
  | "token_limit"
  | "money_balance"
  | "credit_balance"
  | "window_percent";
```

三类来源的含义：

| 来源 | 类型 | 精确度 |
| --- | --- | --- |
| API Key | `api_key` | 已用和剩余 token 精确 |
| 聚合 API / 上游厂商 | `aggregate_api` | 余额精确；按模型剩余 token 为价格估算 |
| OpenAI / ChatGPT 账号池 | `openai_account` | 窗口百分比、重置时间精确；token 仅可选估算 |

### 4.2 价格规则 Model Price Rule

模型价格按规则匹配。每条规则描述一个模型或模型模式在某个计费模式下的价格。

```ts
interface ModelPriceRule {
  id: string;
  provider: "openai" | "anthropic" | "google" | "custom";
  modelPattern: string;
  matchType: "exact" | "prefix" | "glob" | "regex";
  billingMode: "standard" | "batch" | "flex" | "priority" | "custom";
  currency: "USD" | "CNY" | "credit";
  unit: "per_1m_tokens";

  inputPricePer1M: number | null;
  cachedInputPricePer1M: number | null;
  outputPricePer1M: number | null;
  reasoningOutputPricePer1M: number | null;

  cacheWrite5mPricePer1M: number | null;
  cacheWrite1hPricePer1M: number | null;
  cacheHitPricePer1M: number | null;

  longContextThresholdTokens: number | null;
  longContextInputPricePer1M: number | null;
  longContextCachedInputPricePer1M: number | null;
  longContextOutputPricePer1M: number | null;

  source: "official_seed" | "manual" | "supplier_override";
  sourceUrl: string | null;
  seedVersion: string | null;
  enabled: boolean;
  priority: number;
  createdAt: number;
  updatedAt: number;
}
```

匹配优先级：

1. 供应商专属覆盖价。
2. 用户手动精确模型价。
3. 用户手动通配模型价。
4. 官方种子精确模型价。
5. 官方种子通配模型价。
6. 未知价格，展示“成本未知”，不参与剩余 token 估算。

### 4.3 供应商价格配置 Supplier Price Profile

聚合 API 厂商通常不是官方价。需要给每个上游配置余额单位、汇率、倍率和默认计费模式。

```ts
interface SupplierPriceProfile {
  aggregateApiId: string;
  provider: "openai" | "anthropic" | "google" | "custom";
  balanceUnit: "usd" | "cny" | "credit" | "quota";
  balanceToUsdRate: number;
  priceMultiplier: number;
  defaultBillingMode: "standard" | "batch" | "flex" | "priority" | "custom";

  inputRatio: number;
  cachedInputRatio: number;
  outputRatio: number;
  reasoningOutputRatio: number;

  useOfficialSeed: boolean;
  note: string | null;
  updatedAt: number;
}
```

示例：

- 上游返回 USD 余额，官方价计费：`balanceToUsdRate = 1`，`priceMultiplier = 1`。
- 上游返回 CNY 余额，按汇率换算：`balanceToUsdRate = 0.138`。
- 上游价格是官方价 0.8 倍：`priceMultiplier = 0.8`。
- New API 的 quota 余额如果是 `500000 quota = 1 USD`，则 `balanceToUsdRate = 1 / 500000`。

### 4.4 OpenAI 账号计划容量 Account Plan Capacity

OpenAI / ChatGPT 账号池默认只能展示百分比窗口。如果用户希望估算 token，需要允许配置计划容量。

```ts
interface AccountPlanCapacityProfile {
  id: string;
  planKey: string;
  modelPattern: string;
  windowKind: "primary" | "secondary" | "extra";
  windowMinutes: number | null;
  capacityTokens: number | null;
  source: "manual" | "official_seed" | "unknown";
  note: string | null;
  updatedAt: number;
}
```

第一版建议默认不内置账号计划 token 容量，避免误导。UI 可以提供“开启估算”入口，让用户自己配置 Plus / Team / Pro 等计划的窗口容量。

### 6.2 成本计算口径

成本计算输入：

```ts
interface TokenUsageBreakdown {
  inputTokens: number;
  cachedInputTokens: number;
  outputTokens: number;
  reasoningOutputTokens: number;
  totalTokens: number | null;
}
```

第一版建议：

- input 计费 token = `max(inputTokens - cachedInputTokens, 0)`。
- cached input 计费 token = `cachedInputTokens`。
- output 计费 token = `outputTokens`。
- reasoning token 默认只作为 output 的诊断字段，不额外叠加，避免重复计费。
- 如果未来某个协议明确 output 不包含 reasoning，则通过 provider adapter 设置 `reasoningOutputPricePer1M` 并显式加入。

公式：

```text
cost =
  nonCachedInput / 1_000_000 * inputPrice
+ cachedInput / 1_000_000 * cachedInputPrice
+ output / 1_000_000 * outputPrice
```

如果 `cachedInputPrice` 为空，则回退到 `inputPrice`。

### 6.3 剩余 token 估算口径

API Key：

```text
remainingTokens = quotaLimitTokens - usedTokens
```

聚合 API：

```text
balanceUsd = remainingBalance * balanceToUsdRate

effectiveInputPrice = inputPrice * priceMultiplier
effectiveCachedInputPrice = cachedInputPrice * priceMultiplier
effectiveOutputPrice = outputPrice * priceMultiplier
effectiveReasoningPrice = reasoningOutputPriceOrOutputPrice * priceMultiplier

blendedPricePer1M =
  effectiveInputPrice * inputRatio
+ effectiveCachedInputPrice * cachedInputRatio
+ effectiveOutputPrice * outputRatio
+ effectiveReasoningPrice * reasoningOutputRatio

estimatedRemainingTokens = balanceUsd / blendedPricePer1M * 1_000_000
```

OpenAI 账号池：

默认展示：

```text
primaryRemainPercent = 100 - primaryUsedPercent
secondaryRemainPercent = 100 - secondaryUsedPercent
```

如果用户配置了计划容量：

```text
estimatedWindowRemainingTokens = capacityTokens * remainPercent / 100
```

UI 必须标记为“按计划容量估算”。

## 10. 计算与展示口径

### 10.1 精确值

以下值可以标记为“精确”：

- API Key 已用 token。
- API Key 限额剩余 token。
- 请求日志中的模型级 token 用量。
- 聚合 API 返回的余额数值，前提是上游接口可信。
- OpenAI 账号窗口的 used percent、reset time。

### 10.2 估算值

以下值必须标记为“估算”：

- 聚合 API 余额折算的模型级剩余 token。
- 聚合 API 余额折算的混合 token。
- OpenAI 账号池配置计划容量后的 token 剩余。
- 成本估算，尤其是未知 provider、未知缓存语义、未知 reasoning 计费口径时。

### 10.3 不可直接汇总的值

不能直接相加：

- USD 余额和 token 限额。
- OpenAI 账号百分比和聚合 API 金额。
- 不同模型价格下的“可用 token”。

推荐总览展示方式：

```text
API Key：剩余 12.3M tokens（精确）
聚合 API：余额 $83.20，按 gpt-5.4 估算约 18.4M 混合 tokens
OpenAI 账号池：5h 平均剩余 62%，7d 平均剩余 71%
```

## 11. 刷新与缓存策略

### 11.1 聚合 API

- 手动刷新：额度中心点击刷新，调用 `quota/refreshSources`。
- 批量刷新：只刷新启用余额查询的聚合 API。
- 失败处理：保留旧余额，更新 `last_balance_status = error` 和错误信息。
- UI 显示最近成功刷新时间和最近失败原因。

### 11.2 OpenAI 账号池

- 复用现有 usage refresh。
- 允许从额度中心刷新单个账号或全部账号。
- usage refresh 失败不影响聚合 API 刷新。

### 11.3 API Key

- 本地统计，不需要刷新。
- 查询时直接从 `request_token_stats` 和 quota limit 计算。

### 11.4 价格

- 官方种子只在初始化或用户点击“重置官方种子”时写入。
- 不默认联网更新价格。
- 价格修改后需要触发额度中心 query invalidation，重新计算成本与剩余估算。

## 12. 安全与隐私

- 不在额度中心展示任何上游 access token、API key、refresh token。
- `last_balance_json` 如果展示原始内容，需要脱敏。
- 供应商余额查询 token 存储沿用现有 secret 表，不进入普通列表接口。
- 导出价格配置不包含 secret。
- 余额查询错误信息要过滤 Authorization、Bearer、key 等敏感片段。

## 15. 风险与边界

| 风险 | 处理 |
| --- | --- |
| 官方价格变化 | 种子版本化，允许手动更新和覆盖 |
| 上游厂商余额单位不统一 | 供应商 profile 配置 `balanceToUsdRate` |
| output / reasoning 是否重复计费 | 默认 reasoning 只做诊断，不重复加入；provider 可扩展 |
| OpenAI 账号池没有绝对 token | 默认只显示百分比，不估算 |
| 聚合 API 不返回模型级余额 | 按价格估算并标记 |
| 价格缺失导致成本错误 | 显示 `priceStatus = missing`，不参与估算 |
| 余额查询接口失败 | 保留旧值，展示错误状态 |

## 16. 推荐最终口径

额度中心对用户展示时建议使用以下文案原则：

- “API Key 剩余 token”：精确。
- “聚合 API 余额”：上游返回值，精确度取决于上游。
- “聚合 API 按模型预计剩余 token”：估算。
- “OpenAI 账号池剩余”：窗口百分比。
- “OpenAI 账号池预计剩余 token”：仅在用户配置计划容量后展示，且标记为估算。
- “平台总额度”：不展示单一总 token，改为多指标总览。

## 18. 结论

当前代码基础已经足够支撑额度中心。建议按“精确统计 + 明确估算”的方式实现：

- 已用 token：基于本地请求统计，按模型精确。
- API Key 剩余：基于限额，精确。
- 聚合 API 剩余：基于上游余额，金额精确，模型 token 为价格估算。
- OpenAI 账号池：窗口百分比精确，token 剩余需要用户配置容量后估算。
- 模型价格：官方种子开箱即用，但真实运营以手动覆盖和供应商配置为准。
