# Релиз и описание продукта

## Область применения

В этом документе описываются единый вход в выпуск, список продуктов выпуска, параметры ручного запуска и общие пути устранения неполадок текущего склада.

## Единый вход в издательство

На данный момент единственный рабочий процесс публикации:

- `.github/workflows/release-all.yml`
- Интерфейс `dist` будет сначала создаваться отдельно, а затем распространяться на каждую платформу для упаковки и повторного использования заданий.

Метод триггера:

- Поддерживает только `workflow_dispatch`
- По умолчанию не будет автоматически запускаться с помощью push/pull-запроса.

Основной ввод:

- `tag`: Требуется тег выпуска.
- `ref`: построить базовую ветку или зафиксировать, по умолчанию `main`.
- `prerelease`: `auto | true | false`
- Примечание. Рабочий процесс отвечает только за упаковку и публикацию и больше не включает шлюзы тестирования на стороне сервера.

## pre-release Правила

- `prerelease=auto` и `tag` включает `-`: опубликовано как pre-release.
- `prerelease=auto` и `tag` не включают `-`: выпущено как официальная версия.
- `prerelease=true|false`: Принудительная отмена автоматического решения
- При повторном запуске того же `tag` метаданные выпуска будут повторно синхронизированы на основе этих входных данных.

## Локальная запись триггера

Windows Локальный вспомогательный скрипт:

- `scripts/rebuild.ps1`

Общее использование:

```powershell
pwsh -NoLogo -NoProfile -File scripts/rebuild.ps1 `
  -Все платформы `
  -GitRef главная `
  -ReleaseTag v0.1.9 `
  -GithubToken <token>
```

Часто используемые параметры:

- `-AllPlatforms`: Триггер `release-all.yml`
- `-ReleaseTag`: Опубликовать тег
- `-GitRef`: ссылка на сборку
- `-Prerelease`: явно указать статус pre-release.
- `-DownloadArtifacts`: загружать ли артефакты сборки после запуска

## Описание различий платформы

### Windows

- Создание установочной версии и портативной версии одновременно.
- Портативная версия в настоящее время представляет собой единый `exe`, и дополнительный уровень zip включен не будет.

### macOS

- Текущий продукт – `dmg`.
- Поскольку учетная запись Apple Developer не заверена нотариально, первый запуск все равно может быть перехвачен Gatekeeper.
- `dmg` Включено:
  - `Open CodexManager.command`
  - `README-macOS-first-launch.txt`

### Linux

- В настоящее время на настольном компьютере доступны `AppImage` и `deb`.
- Версия Service выпущена в виде сжатого пакета.

## Рекомендуется проверить перед выпуском

1. Подтвердите, что номер версии синхронизирован через `scripts/bump-version.ps1`.
2. Подтвердите, что `Журнал-изменений.md` обновлен.
3. Убедитесь, что сборка пользовательского интерфейса рабочего стола прошла успешно: `pnpm -C apps run build`.
4. Подтвердите, что основные тесты пройдены: `pnpm -C apps run test`, `cargo test --workspace`.
5. Если требуются изменения в протоколе шлюза, дополнительные `scripts/tests/gateway_regression_suite.ps1`

## Устранение распространенных неисправностей

### Отсутствуют артефакты интерфейса

Приоритетные проверки:

- `apps/out/` Можно ли его построить нормально
- Успешно ли завершен этап сборки интерфейса в рабочем процессе.

### Метаданные выпуска неверны.

Приоритетные проверки:

- Включает ли `tag` `-`?
- `prerelease` Является ли автоматическое суждение явно отмененным

### macOS Продукт можно загрузить, но нельзя открыть в первый раз.

Это ожидаемое ограничение в текущем ненотаризованном состоянии и не является сбоем сборки рабочего процесса.

Метод обработки:

1. Сначала перетащите `CodexManager.app` в «Приложения».
2. Дважды щелкните `Open CodexManager.command`.
3. Или выполните:

``` баш
xattr -dr com.apple.quarantine /Applications/CodexManager.app
```
