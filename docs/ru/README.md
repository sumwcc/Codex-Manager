# Описание каталога документов
`docs/` является официальным каталогом документов управления для CodexManager и больше не считается временным хранилищем.

Цель:
- Пусть структурное управление, записи о принятии решений, инструкции по выпуску и руководства по эксплуатации войдут в основную деятельность склада.
- Позвольте новым сотрудникам найти правильный документ, не полагаясь на устную передачу информации.

## Границы ответственности документа
- Корневой каталог `README.md` / `README.en.md`: домашняя навигация, обзор проекта, быстрый запуск.
- Корневой каталог `Журнал-изменений.md`: единственный источник достоверной информации об истории версий и невыпущенных обновлениях.
- `report/*`: Эксплуатация, устранение неполадок, совместимость, руководства по часто задаваемым вопросам.
- `release/*`: Сборка, выпуск, продукт и описание процесса выпуска.
- `docs/plan/*` / `docs/decision/*`: Долгосрочное управление, планы реализации и запись решений.

##Последнее обновление
- Если вы хотите увидеть последний выпущенный контент и невыпущенные обновления основной ветки, сначала прочитайте [Журнал-изменений.md](Журнал-изменений.md).
- Если вы хотите быстро определить, «какой документ наиболее подходит для решения текущей задачи», сначала посмотрите «Главную страницу документа» и «Рекомендуемый ввод» ниже.

## Разделение труда в каталоге

### `docs/plan/`
Используется для сохранения планов реализации, списков управления и поэтапных задач TODO.

### `docs/decision/`
Используется для ведения записей о решениях и ADR.

### `release/`
Используется для сохранения примечаний к выпуску, планов отката, записей о приемке выпуска и создания руководств по выпуску.

### `report/`
Используется для сохранения отчетов о сканировании, устранении неполадок, эксплуатации, совместимости, часто задаваемых вопросов и руководств.

## Отправить правила

### Документация, которая должна быть передана в Git
- Все еще ценно для будущих сотрудников
- Повлияет на последующую разработку, тестирование, выпуск или методы устранения неполадок.
- Может использоваться как часть долгосрочного источника истины для проекта.

### Документацию не рекомендуется отправлять в Git
- Временный проект
- Личные записи процесса
- Одноразовые промежуточные продукты
- Местные протоколы испытаний

## Игнорировать правила
Текущий репозиторий игнорирует следующие документы:
- `docs/**/*.tmp.md`
- `docs/**/*.local.md`

Если это официальный документ, не используйте указанный выше суффикс.

## Предложения по именованию
Рекомендуемый формат:

```текст
Документ долгосрочного хранения: theme.md.
Разовый отчет: ггггММддЧЧммссфффф_topic.md
```

## Соглашение об обслуживании
- При добавлении важных руководящих документов отдавайте приоритет `docs/` и не продолжайте накапливать их в README.
- История версий продолжается в `Журнал-изменений.md`.
- Обзор архитектуры по-прежнему сохраняется в `Архитектура.md`.
- Спецификации сотрудничества по-прежнему сохраняются на уровне `Руководство-для-участников.md`.
— Не записывайте невыпущенные обновления в несколько длинных документов одновременно; если требуется внешнее объяснение, сначала добавьте `Журнал-изменений.md`, и README сохранит только краткое изложение и запись.
